# Допущения MVP

Ниже перечислены ключевые допущения, на которых построена текущая версия решения.

## 1. Формат решения

- Реализован именно `MVP`, а не продакшн-сервис.
- Основной сценарий использования — локальный запуск Python-скрипта.

## 2. Источники данных

- Исходная база блогеров берется из предоставленного Excel-файла.
- Для поиска новых кандидатов используются:
  - локальный демо-датасет;
  - YouTube API как живой внешний источник.

## 3. Ограничения по платформам

- Instagram и Telegram не подключены как полноценные живые источники поиска.
- Это связано с ограничениями доступа, нестабильностью парсинга и отсутствием возможности использовать платные сервисы в MVP.

## 4. Портрет блогера

- Портрет исходной базы строится по упрощенным признакам:
  - username;
  - ключевые слова;
  - заранее заданные базовые темы;
  - базовые визуальные теги;
  - базовый tone of voice.
- В текущей версии нет полного анализа контента исходных профилей, публикаций и реальной вовлеченности.

## 5. Поиск новых кандидатов

- Часть кандидатов — демо-кандидаты, подготовленные для проверки логики пайплайна.
- Часть кандидатов — реальные результаты YouTube-поиска.
- Все кандидаты приводятся к единому формату и затем участвуют в общем ранжировании.

## 6. Персональные офферы

- Офферы генерируются по шаблонной логике на основе результатов матчинга.
- Сообщения персонализируются через совпавшие темы, визуальные теги и tone of voice.
- Прямая отправка сообщений не реализована.

## 7. API и бюджет

- Решение спроектировано так, чтобы работать без обязательных платных API.
- YouTube API используется как бесплатный официальный источник в пределах доступных лимитов.
- Дополнительные LLM API не являются обязательной частью текущей версии.

## 8. Назначение результата

- Решение предназначено для демонстрации подхода, архитектуры и логики автоматизации.
- Текущая версия не позиционируется как инструмент для массовой эксплуатации без доработок.
